OSINT on a Budget: Recon Techniques That Actually Work

Practical OSINT methodology for penetration testers — passive recon, subdomain enumeration, and correlating data without burning through API credits.

2 min read #osint #recon #methodology #tools

Why OSINT Matters Before You Touch a Single Port

Reconnaissance is the phase most people rush. That’s a mistake.

The more you know before sending a single packet, the quieter your footprint — and the better your attack surface map.

Passive Recon Stack

These are the tools I run on every engagement before active scanning:

1# Certificate transparency — free, no auth needed
2curl "https://crt.sh/?q=%25.target.com&output=json" | jq '.[].name_value' | sort -u
3
4# Wayback Machine URLs
5waybackurls target.com | grep "\.php\|\.asp\|admin\|login"
6
7# GitHub dorking
8gh search code "target.com" --language=python

Subdomain Enumeration

Passive first, then active:

1# Passive (no direct contact with target)
2subfinder -d target.com -silent
3
4# DNS bruteforce (active — only when authorized)
5ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt \
6     -u https://FUZZ.target.com -mc 200,301,302

OpSec tip: Use a VPS you don’t care about for active scans. Keep your personal IP out of target logs.

Correlating the Data

Raw lists of subdomains aren’t useful. The goal is finding:

  • Dev/staging environments with weaker controls
  • Forgotten services running old software
  • Infrastructure that reveals cloud provider and region

I pipe everything into a simple SQLite DB and query patterns. More on that in a future post.

Closing Thoughts

Good recon takes time. Budget it properly in your methodology — rushing it is how you miss the easy win that was sitting on a forgotten subdomain the whole time.
